# Cantila Platform Mailboxes — Design

**Date:** 2026-05-29
**Status:** Approved (design)
**Scope:** Let the platform owner ("super user") create real, usable mailboxes on the platform domain `cantila.app` — starting with `info@cantila.app` — from the Console "New mailbox" modal, provisioned for real in the live Mailcow instance (`mail-fsn1`, 178.105.152.116).

## Goal

- `info@cantila.app` exists as a real inbox (IMAP/SMTP/webmail) — the default super-user mailbox.
- The Console "New mailbox" modal lets the owner create additional `cantila.app` mailboxes.
- Mailboxes created this way are provisioned in Mailcow, not just recorded in the control-plane DB.

## Non-goals

- Tenant mailbox provisioning into Mailcow (multi-domain) — stays record-only for now; its own future project.
- Transactional mail **sending** via Mailcow SMTP — that is the separate `MailProvider`/SMTP work (control-plane Mail go-live "todo 5"); this design deliberately does not flip the sending path live.
- Mailcow password reset / first-login flow for created mailboxes (auto-generate + show-once only).

## Key decisions (from brainstorming)

1. **Real provisioning** in Mailcow (not control-plane records only).
2. **Platform project** under the owner account `acc_cantila` owns `cantila.app`; reuses the existing project-scoped mailbox machinery.
3. **Auto-generate password, show once**, never persisted in the control plane.
4. **Separate provisioning port** from the sending port (see A below).
5. **Super user = `useIsOwner()`** (owner email; `cantila-console/src/lib/owner.ts`).
6. `info@cantila.app` is created through the same path once wired (no boot-seed).

## A. Ports: provisioning is separate from sending

Provisioning a mailbox (Mailcow **REST** API) and sending transactional mail (**SMTP** submission) are different mechanisms. Folding both into `MailProvider` would force `mailProvider.live` true and break the password-reset/verify `debugLink` path before SMTP works.

New port `MailboxProvisioner` in `cantila-control-plane/src/mail/provisioner.ts`:

```ts
export interface MailboxProvisioner {
  readonly label: string;
  readonly live: boolean;
  ensureDomain(domain: string): Promise<{ ok: true } | { error: string }>;
  createMailbox(input: {
    address: string;        // full address, e.g. info@cantila.app
    password: string;       // generated by the control plane
    quotaMb: number;
    displayName?: string;
  }): Promise<{ ok: true } | { error: string }>;
  deleteMailbox(address: string): Promise<{ ok: true } | { error: string }>;
}
```

- `StubMailboxProvisioner` — deterministic no-op returning `{ ok: true }`; `live = false`. Keeps offline smoke deterministic.
- `MailcowMailboxProvisioner` (`src/mail/mailcow-provisioner.ts`) — calls Mailcow REST:
  - `ensureDomain` → `GET /api/v1/get/domain/<d>` then `POST /api/v1/add/domain` if absent.
  - `createMailbox` → `POST /api/v1/add/mailbox` (local_part, domain, password, password2, quota, name, active=1).
  - `deleteMailbox` → `POST /api/v1/delete/mailbox` (`["<address>"]`).
  - Auth header `X-API-Key: $MAILCOW_API_KEY`; base URL `$MAILCOW_URL`. `live = true`.
- Env-gated factory `createMailboxProvisioner()` — returns `MailcowMailboxProvisioner` when `MAILCOW_URL` + `MAILCOW_API_KEY` present, else stub. Mirrors `createMailProvider()`.

## B. Data model: Platform project

- Add `platform?: boolean` to the `Project` type (`src/domain/types.ts`), `InMemoryStore`, `PrismaStore`, and a Prisma migration via the existing boot-migration runner (`src/domain/boot-migrations.ts`).
- Boot-seed one system project under `acc_cantila` (pattern mirrors `seed-owner.ts`): id stable (e.g. `proj_platform`), `name: "Platform"`, `platform: true`, associated domain `cantila.app`. Idempotent (upsert).
- `listProjects` (account-facing) filters out `platform` projects so it never appears as a tenant project. Internal accessors can fetch it by id.

## C. Console modal (owner-gated)

In `cantila-console/src/components/MailboxesView.tsx`:
- Read `useIsOwner()`.
- **Owner:** the "New mailbox" modal gains a platform path — a local-part text input with a fixed `@cantila.app` suffix; the Platform project is auto-selected (no project dropdown shown). The button is enabled even when the owner has no tenant projects.
- **Non-owner:** modal unchanged — project dropdown + free-text address, exactly as today.
- After a successful create, the modal switches to a **one-time password panel**: shows the generated password with a copy button and a "store this now, it won't be shown again" notice. Closing returns to the list.
- `api.createHostedMailbox` response type extends to optionally carry `oneTimePassword`.

## D. Flow / password / errors

1. Modal → `api.createHostedMailbox(platformProjectId, { address, quotaMb, displayName })`.
2. Control plane `createHostedMailbox`:
   - Resolve project; if `project.platform`, require the address domain to be `cantila.app` (reject otherwise).
   - Existing checks: valid email, address-not-taken (control-plane store).
   - Generate a strong random password (CSPRNG; reuse `src/auth/tokens.ts` primitives).
   - **Provision first:** `provisioner.ensureDomain("cantila.app")` then `provisioner.createMailbox(...)`. On `{ error }`, return the error and **do not store a record** (no ghost mailboxes).
   - On success, store the hosted-mailbox record (as today) and record the config event.
   - Return `{ ...mailbox, oneTimePassword }` — password returned **only** in this response, never persisted.
3. Non-platform projects: provisioner is **not** called (record-only behavior preserved). Provisioning is gated on `project.platform === true`.

## E. `info@cantila.app`

Created through this exact path once `MAILCOW_API_KEY` + `MAILCOW_URL` are set on the control plane and the Platform project is seeded. No separate boot-seed of the mailbox itself — one code path.

## F. Operational dependency

- Mint a **Mailcow API key** (read-write) in the Mailcow UI (Configuration → Access → API), IP-restricted to the control-plane host. Set `MAILCOW_URL=https://mail.cantila.app` (or the IP until DNS/TLS land) + `MAILCOW_API_KEY` on the control-plane container. Until set, the stub provisioner runs and offline behavior is unchanged.
- The control plane (Coolify host) reaches Mailcow over public HTTPS (port 443) — both are Hetzner FSN1 with public IPs.

## G. Testing

- `scripts/smoke-platform-mailbox.ts`: seeds/loads the Platform project, calls `createHostedMailbox` with `info@cantila.app` against the **stub** provisioner, asserts: domain-validation rejects non-cantila.app on the platform project, success returns a `oneTimePassword`, address-taken errors, and no record is stored on a provisioner error (using a failing fake provisioner). Deterministic, offline.
- `MailcowMailboxProvisioner` verified manually against `mail-fsn1` once the API key is wired (live-only).
- `tsc --noEmit` clean across control-plane and console.
